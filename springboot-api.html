ServerEndpointExporter

ServerEndpointExporter 的作用是在 Spring Boot 环境中，自动扫描和注册使用了 @ServerEndpoint 注解的 WebSocket 服务端点。它是一个实现了 SmartInitializingSingleton 接口的类，会在所有的 Bean 实例化完成后，从 Spring 上下文中获取所有标记了 @ServerEndpoint 的 Bean 的名称，并将它们添加到 ServerContainer 中，从而使客户端可以通过指定的路径连接到这些 WebSocket 服务端点。如果你使用的是外部的容器，例如 Tomcat 或 Jetty，那么你不需要配置这个 Bean，因为这些容器会自动识别和注册 @ServerEndpoint 注解的类。

ApplicationContext

applicationContext是Spring框架中的一个核心接口和容器，它负责管理和配置应用程序中的Bean对象，以及提供一些企业级的服务，例如国际化、资源访问、事件发布、AOP等。applicationContext继承了BeanFactory接口，因此它也支持基本的依赖注入和Bean生命周期管理功能。applicationContext有多种实现类，可以根据不同的场景和需求来选择合适的容器，例如基于XML文件、注解、Java配置等。

以下是一些applicationContext的主要特点和功能：

  • applicationContext可以从不同的来源加载Bean定义，例如XML文件、注解、Java配置等，并且可以支持多个配置文件的合并和覆盖。
  • applicationContext可以解析文本消息，支持国际化和本地化，通过MessageSource接口提供统一的消息访问方式。
  • applicationContext可以访问不同类型的资源，例如类路径、文件系统、URL等，通过ResourceLoader接口提供统一的资源加载方式 。
  • applicationContext可以发布和监听应用程序事件，通过ApplicationEventPublisher接口提供事件发布功能，通过ApplicationListener接口提供事件监听功能 。
  • applicationContext可以支持AOP功能，通过BeanPostProcessor接口提供Bean后置处理功能，通过BeanFactoryPostProcessor接口提供Bean工厂后置处理功能 。

父类HierarchicalBeanFactory：拥有获取父BeanFactory的功能

父类ListableBeanFactory：拥有获取beanNames的功能

父类ResourcePatternResolver：资源加载器，可以一次性获取多个资源（文件资源等等）

父类EnvironmentCapable：可以获取运行时环境（没有设置运行时环境功能）

父类ApplicationEventPublisher：拥有广播事件的功能（没有添加事件监听器的功能）

父类MessageSource：拥有国际化功能

Spring容器最基本的接口就是BeanFactory。BeanFactory负责配置、创建、管理Bean；

ApplicationContext由BeanFactory派生而来；BeanFactory的许多功能需要编程实现，而在ApplicationContext中则可以通过配置的方式实现；

ApplicationContext因此也称之为Spring上下文。Spring容器负责管理Bean与Bean之间的依赖关系。

相关的Aware接口

Spring提供了广泛的Aware回调接口，让bean向容器表明它们需要某种基础设施依赖。

通常Aware有这样一个规则：Aware接口的名称，表示依赖对象的类名称。
例如，一个bean需要使用ApplicationContext，实现ApplicationContextAware接口即可

例如

定义一个bean的类
想要使用ApplicationContext对象，就要实现ApplicationContextAware接口

public class MyApplicationContextAware implements ApplicationContextAware {
    // 用一个全局变量，保存接口回调时传入的applicationContext对象
    private ApplicationContext applicationContext;

    // 使用获取的applicationContext对象
    public void printAllBean(){
        String[] beanDefinitionNames = applicationContext.getBeanDefinitionNames();
        System.out.println("当前容器中的bean如下：");
        for (String beanDefinitionName : beanDefinitionNames) {
            System.out.println(beanDefinitionName);
        }
    }
    @Override
    public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
        this.applicationContext = applicationContext;
    }
}

ApplicationContextAware

ApplicationContextAware的作用是让一个类可以获取它所在的Spring容器，从而可以方便地使用容器中的其他Bean。实现这个接口的类只需要重写setApplicationContext方法，就可以接收到容器对象。这样，就不需要通过new ClassPathXmlApplicationContext()等方式来手动创建容器对象了。

例如，如果我们有一个工具类，需要使用到Spring容器中的某个服务类，我们就可以让这个工具类实现ApplicationContextAware接口，然后在setApplicationContext方法中保存容器对象，再通过getBean方法来获取服务类的实例。这样，我们就可以在工具类中调用服务类的方法了。

以下是一个简单的示例代码：

import org.springframework.beans.BeansException;
import org.springframework.context.ApplicationContext;
import org.springframework.context.ApplicationContextAware;

public class MyUtil implements ApplicationContextAware {

    private static ApplicationContext context;

    @Override
    public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
        // 将容器对象保存到静态变量中
        context = applicationContext;
    }

    public static void doSomething() {
        // 从容器中获取服务类的实例
        MyService myService = context.getBean(MyService.class);
        // 调用服务类的方法
        myService.doSomething();
    }
}

BeanPostProcessor

public interface BeanPostProcessor {

    Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException;

    Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException;

}

BeanPostProcessor是一个Spring框架提供的接口，它的作用是在bean实例化之后，初始化前后对bean进行一些额外的处理，比如修改属性值，添加代理对象，增加功能等。BeanPostProcessor可以应用于容器中的所有bean，也可以通过注解或者XML配置指定某些bean使用特定的BeanPostProcessor。

BeanPostProcessor接口有两个方法：postProcessBeforeInitialization和postProcessAfterInitialization，分别在bean的初始化方法（如@PostConstruct或者init-method）之前和之后执行。这两个方法都接收一个bean对象和一个beanName作为参数，返回一个处理后的bean对象。

演示了如何使用BeanPostProcessor来给容器中的所有bean添加一个前缀：

// 定义一个普通的bean
@Component
public class Hello {
    private String name;

    public void setName(String name) {
        this.name = name;
    }

    public void sayHello() {
        System.out.println("Hello, " + name);
    }
}

// 定义一个BeanPostProcessor的实现类
@Component
public class PrefixBeanPostProcessor implements BeanPostProcessor {
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
        // 在初始化之前，给所有bean的name属性添加一个前缀
        if (bean instanceof Hello) {
            Hello hello = (Hello) bean;
            hello.setName("Mr. " + hello.getName());
        }
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
        // 在初始化之后，不做任何处理
        return bean;
    }
}

// 测试类
public class Test {
    public static void main(String[] args) {
        // 创建一个Spring容器
        ApplicationContext context = new AnnotationConfigApplicationContext("com.example");
        // 从容器中获取Hello这个bean
        Hello hello = context.getBean(Hello.class);
        // 调用sayHello方法
        hello.sayHello(); // 输出：Hello, Mr. Bob
    }
}

通过BeanPostProcessor，我们可以在不修改原始bean代码的情况下，对bean进行一些自定义的操作。这是一种非常灵活和强大的机制，Spring框架本身也使用了很多BeanPostProcessor来实现一些核心功能，比如AOP、事务、注解等。

package com.dpb.processor;

import org.springframework.beans.BeansException;
import org.springframework.beans.factory.config.BeanPostProcessor;
/**
 * 自定义BeanPostProcessor实现类
 * BeanPostProcessor接口的作用是：
 *      我们可以通过该接口中的方法在bean实例化、配置以及其他初始化方法前后添加一些我们自己的逻辑
 * @author dengp
 *
 */
public class MyBeanPostProcessor implements BeanPostProcessor{

    /**
     * 实例化、依赖注入完毕，在调用显示的初始化之前完成一些定制的初始化任务
     * 注意：方法返回值不能为null
     * 如果返回null那么在后续初始化方法将报空指针异常或者通过getBean()方法获取不到bena实例对象
     * 因为后置处理器从Spring IoC容器中取出bean实例对象没有再次放回IoC容器中
     */
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
        System.out.println("before--实例化的bean对象:"+bean+"\t"+beanName);
        // 可以根据beanName不同执行不同的处理操作
        return bean;
    }

    /**
     * 实例化、依赖注入、初始化完毕时执行 
     * 注意：方法返回值不能为null
     * 如果返回null那么在后续初始化方法将报空指针异常或者通过getBean()方法获取不到bena实例对象
     * 因为后置处理器从Spring IoC容器中取出bean实例对象没有再次放回IoC容器中
     */
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
        System.out.println("after...实例化的bean对象:"+bean+"\t"+beanName);
        // 可以根据beanName不同执行不同的处理操作
        return bean;
    }

}

如果BeanPostProcessor的实现类依赖于其他bean，那么这些bean也必须在其他bean创建之前就被创建，所以它们也无法被AOP代理。

这就导致了一个限制，我们无法对BeanPostProcessor的实现类或者它依赖的bean使用AOP织入通知，也无法对它们应用其他BeanPostProcessor提供的功能。

InitializingBean

  • InitializingBean是一个Spring框架提供的接口，它的作用是在bean实例化之后，初始化前后对bean进行一些额外的处理，比如修改属性值，添加代理对象，增加功能等。
  • InitializingBean接口只有一个方法：afterPropertiesSet，它会在bean的所有属性被设置之后调用，可以用来检查属性是否正确，或者执行一些自定义的初始化逻辑。
  • InitializingBean接口和init-method属性的关系是：如果一个bean既实现了InitializingBean接口，又配置了init-method属性，那么Spring容器会先调用afterPropertiesSet方法，然后再调用init-method指定的方法。
  • InitializingBean接口的实现方式是：通过反射机制，在AbstractAutowireCapableBeanFactory类中的initializeBean方法中，判断bean是否是InitializingBean的实例，如果是，则调用afterPropertiesSet方法。
// 定义一个User类，实现InitializingBean接口
public class User implements InitializingBean {
    private String name;
    private int age;

    // 构造方法
    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // 重写afterPropertiesSet方法，在bean的属性设置完成后执行一些逻辑
    @Override
    public void afterPropertiesSet() throws Exception {
        // 检查name属性是否为空，如果为空则抛出异常
        if (name == null || name.isEmpty()) {
            throw new IllegalArgumentException("Name cannot be null or empty");
        }
        // 检查age属性是否合法，如果不合法则抛出异常
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("Age must be between 0 and 150");
        }
        // 打印一些信息
        System.out.println("User bean initialized by InitializingBean: " + this);
    }

    // 定义一个自定义的初始化方法
    public void init() {
        // 给name属性添加一个前缀
        name = "Mr. " + name;
        // 给age属性加10
        age = age + 10;
        // 打印一些信息
        System.out.println("User bean initialized by init-method: " + this);
    }

    // 重写toString方法，方便打印
    @Override
    public String toString() {
        return "User{" +
                "name='" + name + '\'' +
                ", age=" + age +
                '}';
    }
}

// 定义一个配置类，用来创建User bean
@Configuration
public class UserConfig {

    // 使用@Bean注解，指定bean的名称、类型和init-method属性
    @Bean(name = "user", initMethod = "init")
    public User createUser() {
        // 创建一个User对象，传入name和age参数
        User user = new User("Alice", 20);
        // 返回User对象
        return user;
    }
}

// 定义一个测试类，用来测试User bean的创建和初始化过程
public class UserTest {

    public static void main(String[] args) {
        // 创建一个Spring容器，传入配置类的字节码对象
        ApplicationContext context = new AnnotationConfigApplicationContext(UserConfig.class);
        // 从容器中获取User bean，指定bean的名称和类型
        User user = context.getBean("user", User.class);
        // 打印User对象
        System.out.println(user);
    }
}

运行结果：

User bean initialized by InitializingBean: User{name='Alice', age=20}
User bean initialized by init-method: User{name='Mr. Alice', age=30}
User{name='Mr. Alice', age=30}

可以看到，当Spring容器创建User bean时，会自动调用afterPropertiesSet方法和init方法，在这两个方法中执行了一些初始化逻辑，比如检查属性是否正确，修改属性值等。这样就实现了在bean实例化之后，初始化前后对bean进行一些额外的处理的功能。

注册必要的destruction相关回调接口

  • 注册必要的destruction相关回调接口是指在Spring bean的生命周期中，为了保证在应用上下文被销毁之前，能够执行一些自定义的销毁逻辑，需要向Spring容器注册一些回调方法，让容器在合适的时机调用它们。
  • 注册destruction相关回调接口的方式有以下几种：
    • 实现DisposableBean接口，该接口只有一个destroy方法，在需要进行销毁前置的类中实现该方法即可。Spring容器会在释放该bean之前调用该方法。
    • 配置自定义的destroy-method属性，该属性可以在XML配置文件中或者@Bean注解中指定，它的值是一个方法名，该方法必须是bean类中的一个公共方法，且无参数和返回值。Spring容器会在释放该bean之前调用该方法。
    • 使用@PreDestroy注解，该注解可以标注在bean类中的任意一个公共方法上，且无参数和返回值。Spring容器会在释放该bean之前调用该方法。
  • 注册destruction相关回调接口的作用是：
    • 可以释放一些资源，比如关闭数据库连接，关闭文件流，关闭线程池等。
    • 可以执行一些清理操作，比如清除缓存，清除临时文件，清除状态信息等。
    • 可以执行一些通知操作，比如发送消息，记录日志，发送邮件等。

DisposableBean

  • DisposableBean接口是一个Spring框架提供的接口，它的作用是在bean销毁之前执行一些自定义的逻辑，比如释放资源，清理缓存，发送通知等。
  • DisposableBean接口只有一个方法：destroy，它会在容器释放该bean之前调用，可以用来执行一些销毁前置的操作。
  • DisposableBean接口和destroy-method属性的关系是：如果一个bean既实现了DisposableBean接口，又配置了destroy-method属性，那么Spring容器会先调用destroy方法，然后再调用destroy-method指定的方法。
  • DisposableBean接口的实现方式是：通过反射机制，在AbstractBeanFactory类中的destroyBean方法中，判断bean是否是DisposableBean的实例，如果是，则调用destroy方法。

下面是一个简单的例子，演示了如何使用DisposableBean接口来实现bean的销毁逻辑：

// 定义一个User类，实现DisposableBean接口
public class User implements DisposableBean {
    private String name;
    private int age;

    // 构造方法
    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // 重写destroy方法，在bean销毁之前执行一些逻辑
    @Override
    public void destroy() throws Exception {
        // 打印一些信息
        System.out.println("User bean destroyed: " + this);
        // 释放一些资源，比如关闭数据库连接
        // closeConnection();
    }

    // 重写toString方法，方便打印
    @Override
    public String toString() {
        return "User{" +
                "name='" + name + '\'' +
                ", age=" + age +
                '}';
    }
}

// 定义一个配置类，用来创建User bean
@Configuration
public class UserConfig {

    // 使用@Bean注解，指定bean的名称和类型
    @Bean(name = "user")
    public User createUser() {
        // 创建一个User对象，传入name和age参数
        User user = new User("Alice", 20);
        // 返回User对象
        return user;
    }
}

// 定义一个测试类，用来测试User bean的创建和销毁过程
public class UserTest {

    public static void main(String[] args) {
        // 创建一个Spring容器，传入配置类的字节码对象
        AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(UserConfig.class);
        // 从容器中获取User bean，指定bean的名称和类型
        User user = context.getBean("user", User.class);
        // 打印User对象
        System.out.println(user);
        // 关闭容器
        context.close();
    }
}

运行结果：

User{name='Alice', age=20}
User bean destroyed: User{name='Alice', age=20}

可以看到，当Spring容器关闭时，会自动调用destroy方法，在该方法中执行了一些销毁逻辑，比如打印信息等。这样就实现了在bean销毁之前执行一些自定义的逻辑的功能。

@Bean指定destroy-method属性

// 定义一个User类
public class User {
    // 省略其他属性和方法
    // 定义一个初始化方法
    public void init() {
        // 在这里写初始化逻辑
    }
    // 定义一个销毁方法
    public void destroy() {
        // 在这里写销毁逻辑
    }
}

// 定义一个配置类，用来创建User bean
@Configuration
public class UserConfig {

    // 使用@Bean注解，指定bean的名称、类型、init-method和destroy-method
    @Bean(name = "user", initMethod = "init", destroyMethod = "destroy")
    public User createUser() {
        // 创建一个User对象
        User user = new User();
        // 返回User对象
        return user;
    }
}

当Spring容器创建User bean时，会自动调用init方法，在该方法中执行一些初始化逻辑。当Spring容器关闭时，会自动调用destroy方法，在该方法中执行一些销毁逻辑。

@PreDestroy

@PreDestroy是一个注解，它的作用是在一个对象被销毁之前执行一些操作，比如释放资源或清理缓存。它可以用来修饰一个非静态的void方法，而且这个方法不能有抛出异常声明。它通常和@PostConstruct一起使用，@PostConstruct是在对象被创建之后执行一些初始化操作。这两个注解都是Java EE 5规范中引入的，需要支持Servlet 2.5规范的服务器才能使用。

import javax.annotation.PostConstruct;
import javax.annotation.PreDestroy;

public class DemoBean {

    public DemoBean() {
        System.out.println("DemoBean 构造方法");
    }

    @PostConstruct
    public void init() {
        System.out.println("DemoBean @PostConstruct 方法");
    }

    @PreDestroy
    public void destroy() {
        System.out.println("DemoBean @PreDestroy 方法");
    }

    public void doSomething() {
        System.out.println("DemoBean doSomething 方法");
    }
}

在Spring Boot中，可以使用@Bean注解来注册这个类为一个bean，然后在应用程序关闭时，会自动调用@PreDestroy方法。

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.context.annotation.Bean;

@SpringBootApplication
public class DemoApplication {

    public static void main(String[] args) {
        ConfigurableApplicationContext context = SpringApplication.run(DemoApplication.class, args);
        DemoBean demoBean = context.getBean(DemoBean.class);
        demoBean.doSomething();
        context.close();
    }

    @Bean
    public DemoBean demoBean() {
        return new DemoBean();
    }
}

运行结果如下：

DemoBean 构造方法
DemoBean @PostConstruct 方法
DemoBean doSomething 方法
DemoBean @PreDestroy 方法

可以看到，在构造方法之后执行了@PostConstruct方法，在关闭容器之前执行了@PreDestroy方法。

AnnotationConfigApplicationContext

AnnotationConfigApplicationContext的作用是：

  • 它是一个基于注解的Spring应用程序上下文，可以管理Spring Bean的生命周期和依赖注入。
  • 它可以通过注册带有@Configuration注解的Java配置类，来加载和解析配置信息，创建和注册Bean对象。
  • 它也可以通过扫描指定的包路径，来自动发现和注册带有@Component注解或其派生注解（如@Service, @Repository, @Controller等）的Bean对象。
  • 它还可以通过@Import注解，来导入其他的配置类或者普通的Java类，实现模块化的配置。
  • 它提供了一些便利的方法，如refresh(), close(), getBean(), publishEvent()等，来操作和控制应用程序上下文。

AnnotationConfigApplicationContext可以用来创建和配置基于注解的Spring应用程序，而不需要使用XML文件。它可以提高开发效率，增强代码可读性，减少配置错误。使用有两种，一种是指定java文件配置类，另一种是指定扫描路径。

承载bean定义的java文件本质上与其他java文件没有任何区别，只是我们需要一些spring的标签来标识它，这样spring就知道从这个文件中加载bean。

    }
}

先标识了一个简单的CD java bean，内部没有任何属性。然后是Player java bean，内部引用指向一个CD实例，我们通过方法的参数来让spring注入依赖。

最后测试下，上下文选择AnnotationConfigApplicationContext，传入配置类的类实例来告诉spring配置类是哪一个。

public class Main {
    public static void main(String args[]) {
        ApplicationContext applicationContext = new AnnotationConfigApplicationContext(JavaConfig.class);
        Player2 player2 = (Player2) applicationContext.getBean("player2");
        player2.play();
    }

SpringApplicationBuilder

SpringApplicationBuilder的作用是：

  • 它是一个用于构建Spring应用程序的构建器类。
  • 它提供了一种简单的方式来配置和构建 Spring 应用程序，包括设置应用程序上下文、添加配置类、设置环境、设置属性等。
  • 它具有便利的流利的 API 和上下文层次结构支持。
  • 它可以与SpringApplication类配合使用，或者替代SpringApplication类，来从Java main方法引导Spring应用。

SpringApplicationBuilder可以用来创建和配置不同类型的Spring应用程序，例如Web应用、控制台应用、批处理应用等。它可以实现多个应用程序上下文的嵌套和继承，以实现模块化和分层的设计。

// 定义一个父配置类，使用@Configuration注解标注
@Configuration
public class ParentConfig {

    // 使用@Bean注解定义一个Bean对象，方法名为Bean的ID
    @Bean
    public String hello() {
        // 返回一个字符串对象
        return "Hello";
    }
}

// 定义一个子配置类，使用@Configuration注解标注
@Configuration
public class ChildConfig {

    // 使用@Bean注解定义一个Bean对象，方法名为Bean的ID
    @Bean
    public String world() {
        // 返回一个字符串对象
        return "World";
    }
}

// 定义一个运行类，包含主方法
public class Application {

    public static void main(String[] args) {
        // 创建一个SpringApplicationBuilder对象，传入父配置类作为参数
        SpringApplicationBuilder builder = new SpringApplicationBuilder(ParentConfig.class);
        // 设置Web类型为NONE，表示不启动任何Web容器或服务器
        builder.web(WebApplicationType.NONE);
        // 创建并运行一个父应用程序上下文，返回一个ConfigurableApplicationContext对象
        ConfigurableApplicationContext parentContext = builder.run(args);
        // 从父上下文中获取hello类型的Bean对象
        String hello = parentContext.getBean("hello", String.class);
        // 打印出hello对象的值
        System.out.println(hello);

        // 创建一个新的SpringApplicationBuilder对象，传入子配置类作为参数
        SpringApplicationBuilder childBuilder = new SpringApplicationBuilder(ChildConfig.class);
        // 设置Web类型为NONE，表示不启动任何Web容器或服务器
        childBuilder.web(WebApplicationType.NONE);
        // 设置父上下文为parentContext对象
        childBuilder.parent(parentContext);
        // 创建并运行一个子应用程序上下文，返回一个ConfigurableApplicationContext对象
        ConfigurableApplicationContext childContext = childBuilder.run(args);
        // 从子上下文中获取world类型的Bean对象
        String world = childContext.getBean("world", String.class);
        // 打印出world对象的值
        System.out.println(world);

        // 关闭子上下文和父上下文
        childContext.close();
        parentContext.close();
    }
}
